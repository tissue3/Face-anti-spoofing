3.3 移植C++代码至android平台


我们课题的目的是将人脸识别防伪技术应用于android手机，这意味着我们需要封装一个apk。以下是我们实现此目标的全过程。

第一步，改写C++代码，删去冗长的部分。

我们不需要在android手机上再对80张照片进行特征提取，这对手机arm内核的CPU是一个考验。通常这将比在电脑上提取特征电脑上多出3-5倍的时间（reference）。更重要的是80张照片占据了9.53 MB的存储空间，而一个存储已提取的特征的文档仅仅需要23.9 KB的内存。同时，在我们的测试中发现机器学习过程因为调用了opencv宏包中的函数，所用时长比80张特征提取要短 秒。因此更好选择是将提取好的特征写入记事本文档，仅仅让android手机处理机器学习和针对一张照片的特征提取，这将大大节省所需时长和应用占据的内存。

第二步，安装所需工具。

以下所有这些工具的平带都是windows 64-bit。

OpenCV，前面已经介绍过，是一个基于（开源）发行的跨平台计算机视觉库。但是现在，我们的平台从windows 64-bit变为了android 5.1.1，需要的不再是OpenCV for windows，而是OpenCV for android的sdk，版本需对应着windows平台下的版本，即2.4.9。这样才能在android平台下读取和应用其中的库函数。

第三步，配置环境变量

下载，安装好以上的工具后，我们还需要配置环境变量。环境变量是一个具有特定名字的对象，它包含了一个或者多个应用程序所将使用到的信息。例如path，当要求系统运行一个程序而没有告诉它程序所在的完整路径时，系统除了在当前目录下面寻找此程序外，还应到path中指定的路径去找。

如表3.3.1所示，除了Android Studio，其余各工具都需要进行环境变量的配置。因为我们使用的是Android Studio，这里Android SDK不需要在电脑的环境变量里配置，而是在打开Android Studio时直接在Configure-Project Defaults-Project Structure中进行修改。同时JDK，NDK也需要在此处配置。

除了Android Studio中，JDK和NDK还需要在电脑的环境变量中配置。这是为了保证我们使用javah命令创建.h文件，NDK-build命令生成so动态库的时候计算机能够根据环境变量找到工具。

第四步，写接口函数。

配置好环境变量，现在开始C++源生代码的移植。Android的平台框架图如图3.3.5所示，Android上层的Application和Application Framework都是使用Java编写，底层包括系统和使用众多的Libraries都是C/C++编写的，内核是Linux。所以上层Java要调用底层的C/C++函数库必须通过Java的JNI来实现。NDK的作用是将C/C++代码编译封装成.so动态链接库。

图3.3.5 Android平台的框架图 (http://www.dzrmcu.com/shownews.asp?id=305)

先建立一个android应用项目，在项目目录下找到 app-src-main，在此文件夹下新建一个jni文件夹。这个文件夹将用于存放所有所需的C/C++的源生代码。在jni中新建一个cpp文件，写入以下代码。

extern "C" {
JNIEXPORT jfloat JNICALL Java_com_facere1_code_facere1_MainActivity_ImgFun(
JNIEnv* env, jclass obj, jstring buf, jstring buf1) {
const char* str;
const char* str1;
str = env->GetStringUTFChars(buf, false);
str1 = env->GetStringUTFChars(buf1, false);
float fret = mainProc((char*)str, (char*)str1);
env->ReleaseStringUTFChars(buf, str);
env->ReleaseStringUTFChars(buf1, str1);
return fret;
}

}

这里介绍接口函数的语法。extern ”C”{}内写入连接Java层和JNI层的接口函数。先申明接口函数。JNIEXPORT后写输出变量类型，JNICALL后写Mainactivity中Acitivity的reference和输入函数名称，用短线“_”连接。括号中JNIEnv *, jclass是必须保留的，“,”之后是输入变量类型和变量名称。所有的输入变量类型都要加上 “j”，代表java。

除了语法，还需要注意java和C/C++的字符串并不是完全一样。GetStringUTFChars的作用是将传进来的prompt（jstring类型）转换成为UTF-8的格式，使之能够在本地方法中使用。使用完转换之后的对象后，需要显示调用ReleaseStringUTFChars方法，让JVM释放转换成UTF-8的string的对象的空间，如果不显示的调用的话，JVM中会一直保存该对象，导致内存溢出。最后，mainProc()就是C++源生代码的主函数，其余代码和源生代码完全一致。（http://www.cnblogs.com/hoys/archive/2011/09/30/2196290.html）

在同一个目录下，添加一个空的c文件，这是因为Android Studio对于NDK的兼容性还不够好，要不然gradle 编译android代码的时候会报错。

写好了JNI这头接口函数，在Java端，我们在MainActiviy里面同样需要申明接口函数。即添加如下代码。

public static native float ImgFun(String imgPath,String textPath);

第五步，编译jni方法头文件

这里，我们使用javah自动生成jni方法对应的头文件，这种方便比直接写代码不容易出错。在AS提供的Teminal或者cmd.exe命令行程序中输入javah -d %输出.h文件的路径% -classpath %编译的文件路径名% %文件名%。比如说我们的程序所用的命令就是 javah –d D:\temporary\Facere1\app\src\main\jni –classpath D:\temporary\Facere1\app\src\main\java com.facere1.code.facere1.MainActivity

第六步，编译make文件。

这之后，在jni中新建2个make文件，Android.mk 和 Application.mk，它们保存着如何编译的配置信息，即指导编译器如何来编译程序，并决定编译的结果是什么。下面这段代码是我们的Android.mk的内容。

LOCAL_PATH := $(call my-dir)

include $(CLEAR_VARS)

OPENCV_LIB_TYPE:=STATIC

ifeq ("$(wildcard $(OPENCV_MK_PATH))","")


include D:\TOOL\OpenCV-2.4.9-android-sdk\sdk\native\jni\OpenCV.mk

else

include $(OPENCV_MK_PATH)

endif


LOCAL_MODULE := ImgFun

LOCAL_SRC_FILES :=ImgFun.cpp

LOCAL_LDLIBS += -lm -llog

include $(BUILD_SHARED_LIBRARY)

具体的语法我们在这里不予介绍，请参考http://android.mk/的官方指南。请注意，include后加入OpenCV for Android中make文件的路径，只有这样我们才能调用OpenCV的库函数。LOCAL_MODULE中ImgFun指的是.so动态库，而LOCAL_SRC_FILES中ImgFun.cpp指的是jni文件夹中待编译的文件本身。

在Application.mk文件中写入如下代码。

APP_STL := gnustl_static

APP_CPPFLAGS := -frtti -fexceptions

APP_ABI := armeabi-v7a

APP_PLATFORM := android-22

默认情况下，NDK的编译系统会为"armeabi"ABI生成机器代码。这里，为了在ARMv7的设备上支持硬件FPU指令我们可以使用了APP_ABI := armeabi-v7a 。(http://www.iteye.com/topic/1113483)

第七步，生成和调用.so动态库

现在，在Teminal或者cmd.exe命令行程序中使用cd命令，进入jni文件目录，然后再命令行中输入ndk-build命令，这样就生成了.so动态库。

图3.3.6 通过ndk生成的.so动态库

最后，我们在java层调用这个动态库，需要在MainActivity写入如下代码。loadLibrary()这个方法调用了按照make文件编译生成的ImgFun.so动态库。

static {
System.loadLibrary("ImgFun");
}

至此，C++代码在android平台的移植就算完成了。


3.4 android功能要求以及程序界面设计

这个android应用程序的功能是判断一张图片是否是真人。第一步，读取图片。图片有两种，其一是通过android手机自带摄像头拍摄所得照片；其二则是从本地相册中选取的照片。第二步，显示读取的照片。第三步，判断此图片是真人还是照片，仅需调用C++源生代码。

由如上的功能程序主界面需要两个按钮，一个图片显示区和一个判断结果显示区，如图3.4.1所示。

图3.4.1 应用程序基础界面

添加完基本元素以后，此代码的基本功能较为简单，在此不进行赘述。以下提几个值得注意的细节。

第一，我们通过“从相册中选择”按钮和“拍照”按钮分别复写各自的onClick方法，但是拍照所得照片读入后需要我们调用Environment.getExternalStorageDirectory()方法存入sd卡上，保证用户做好此次判断后还能找回这张照片。

第二，通过相机获取图片的时候还需要注意，uri的返还值可能是空的，因为图片是封装到bundle中传递的，有些机器可以获取有些则不能。对于不能获取uri的机器，需要我们用getExtras方法获取bundle，之后通过(Bitmap) bundle.get("data")获取到相机图片的bitmap数据。最后通过compress方法将bitmap的uri返还，将压缩率设为100就不会失真。

当我们用各机型手机测试相册读取图片的时候，又发现另一个漏洞: 不是所有机型的手机也都能通过getData()方法直接获得图片路径。

if(data.getData().toString().contains("file")){
String path = data.getData().getPath();

}else{

String path = getPath(data.getData());

}

private String getPath(Uri uri){
String[] proj = {MediaStore.Images.Media.DATA};
Cursor cursor = getContentResolver().query(uri,proj,null,null,null,null);
int column_index = cursor.getColumnIndexOrThrow(MediaStore.Images.Media.DATA);
cursor.moveToFirst();
String path = cursor.getString(column_index);
cursor.close();
return path;
}

以上的代码用于解决此问题。这里data是intent的一个对象。当data.getData()获得的是uri仅仅是一个索引值的时候，我们就需要根据索引值获得正确的图片路径。

显示图片的时候，如果我们不压缩图片，不同机型的android手机分辨率不同，可能会出现内存溢出的情况。此时，我们重新设置Options对象的inSampleSize，即压缩比。实际比例依据是主流手机的800*480分辨率。(reference) 压缩后的图片会出现一定程度的失真，但是以一般手机屏幕的辨析度来说并无大碍。

注意以上几点，此应用程序在一般android手机上就能够成功运行了。
